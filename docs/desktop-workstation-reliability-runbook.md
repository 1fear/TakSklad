# Desktop Workstation Reliability Runbook

Документ для поддержки рабочих Windows-компьютеров склада после reliability-обновления desktop-приложения.

Не читать и не пересылать секреты, `.env`, Google credentials, Telegram токены, реальные клиентские выгрузки, backups и outputs. Для поддержки использовать только UI, диагностический пакет и redacted логи.

## 1. Устаревшая Сборка

Признаки:

- внизу окна видно старую версию или статус блокировки обновления;
- приложение пишет, что требуется обновление;
- сканирование/операции заблокированы до обновления.

Действия:

1. Не удалять `TakSklad_data.json` и локальные очереди.
2. Открыть диагностический пакет через кнопку `ДИАГНОСТИКА`.
3. Проверить строку версии: local version, latest/min-supported, package type, workstation id.
4. Установить свежий Windows archive/release по штатному runbook.
5. После запуска проверить, что статус версии больше не заблокирован.

Если manifest/update server недоступен, приложение должно показать статус “проверка обновлений недоступна”, а не скрывать проблему.

## 2. Битые Локальные Данные

Признаки:

- ошибка чтения локального JSON;
- пустой список после аварийного закрытия;
- пропали pending saves/prints/Telegram/backend events в UI.

Действия:

1. Не удалять рабочую папку приложения.
2. Перезапустить TakSklad один раз: приложение пробует восстановить `TakSklad_data.json` из last-good backup.
3. Если восстановление не помогло, создать диагностический пакет.
4. Проверить только counts очередей, не открывать raw queue JSON без необходимости.
5. Перед ручной правкой сделать копию всей папки приложения.

Ожидаемо: backup/restore не должен логировать токены, КИЗы, клиентов и адреса.

## 3. Очереди И Зависшие Отправки

Признаки:

- блок `Синхронизация` показывает pending/blocked;
- есть очередь Google saves, backend scans/completes, печати или Telegram;
- backend 409 после скана.

Действия:

1. Открыть кнопку `ОЧЕРЕДИ`.
2. Смотреть counts, oldest age, attempts и blocked reason.
3. Нажать retry только если UI разрешает.
4. Если событие blocked, не считать его успешно отправленным.
5. Для backend 409 по КИЗу проверить текст: owner found, owner unknown, wrong SKU или reusable-after-return.

Rejected/no-order scans не должны попадать в backup/backend/pending queues.

## 4. Сеть, Backend, Google И Manifest

Признаки:

- список не обновляется;
- backend недоступен;
- Google Sheets quota/timeout;
- manifest/update check unavailable.

Действия:

1. Создать диагностический пакет кнопкой `ДИАГНОСТИКА`.
2. Проверить probe class: `dns`, `tls`, `auth_rejected`, `backend_unavailable`, `unknown`.
3. Если есть cached orders, можно работать только с последним загруженным списком, но не считать синхронизацию успешной.
4. После восстановления сети открыть `ОЧЕРЕДИ` и повторить retry.

Диагностический пакет не должен содержать raw log lines, КИЗы, токены, chat_id, client/address или `.env`.

## 5. Сбой Обновления

Признаки:

- обновление не завершилось;
- приложение не стартует после updater;
- появились папки `new`/`previous`/update log.

Действия:

1. Не удалять runtime files и очереди.
2. Проверить update log и диагностический пакет.
3. Если updater не смог заменить сборку, восстановить `previous` или установить свежий archive вручную.
4. Проверить, что `TakSklad_data.json`, pending queues, settings, logs и backups не перезаписаны новой сборкой.
5. Не запускать старый exe по кругу, если status говорит о forced update.

## 6. Возвраты

Правила:

- полный возврат остается штатным сценарием;
- partial selection не должен сохраняться как полный возврат;
- если частичный возврат не поддержан, оператор видит сообщение и запись не выполняется;
- already-returned заказ показывает дату возврата/основание и кнопка возврата disabled.

Сообщение для unsupported partial:

```text
Частичный возврат пока не поддержан для этого пути. Выберите все позиции или проведите возврат вручную через backend/SkladBot.
```

КИЗ становится reusable только после успешного backend return/undo/reset movement. Failed/unsupported return не освобождает КИЗ.

## 7. Сканирование И Обновление На Двух ПК

Правила:

- поле сканирования disabled до выбора позиции;
- wrong SKU, duplicate и no-order scan не пишут очередь;
- если другой ПК уже закрыл/вернул позицию, refresh отключает сканирование текущей позиции;
- если другой ПК добавил КИЗ, refresh мержит remote saved codes и local unsaved без double count.

Если refresh говорит, что позиция больше не активна, не продолжать сканирование этой позиции. Нужно обновить список и выбрать актуальную позицию.

## 8. Windows Manual Acceptance

Автотесты не заменяют проверку на реальном рабочем ПК.

Перед релизом на складе проверить и зафиксировать:

1. Windows app стартует с актуальной версией и workstation id.
2. Второй запуск блокируется и не открывает второе рабочее окно.
3. Сканер вводит КИЗ в выбранную позицию.
4. До выбора позиции scan entry disabled.
5. Wrong SKU показывает понятное сообщение и не пишет очередь.
6. Дубль КИЗа показывает owner/reuse reason.
7. Возврат full проходит, partial показывает unsupported message.
8. `ОЧЕРЕДИ` показывает pending/blocked counts и retry работает только для retryable.
9. `ДИАГНОСТИКА` создает redacted JSON bundle.
10. Обновление/rollback не удаляет runtime queues.
11. Печать сводного листа работает на реальном принтере.
12. Два ПК: refresh во время сканирования не теряет local unsaved scan и блокирует позицию, если она закрыта на другом ПК.

В evidence записать дату, версию, workstation id hash/label, сценарий, результат и оставшийся риск. Не прикладывать raw КИЗы, токены, клиентские выгрузки или credentials.
