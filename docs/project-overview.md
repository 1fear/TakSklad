# Обзор Проекта TakSklad

TakSklad - Python/Tkinter-приложение для склада, которое сейчас работает как локальная точка управления сканированием и обработкой документов. Приложение запускается на компьютере склада, работает с Excel-файлами и Google Sheets, принимает ввод со сканера, печатает сводные листы, хранит очереди восстановления, создает отчеты и отправляет рабочие файлы через Telegram.

Долгосрочная цель - развивать TakSklad как WMS-экосистему, а не только как отдельную desktop-программу. Экосистема должна помогать складу обрабатывать документы, контролировать сканы, уменьшать ручную работу, раньше находить ошибки, сохранять историю и делать процессы прозрачнее.

## Текущий Продукт

Сейчас приложение закрывает такие задачи:

- импорт Excel в Google Sheets;
- импорт Excel-файлов, отправленных в Telegram-бот;
- группировка заказов для сканирования;
- ввод КИЗов через сканер;
- запись отсканированных КИЗов в Google Sheets;
- прямая печать сводных листов;
- локальные backup-файлы и очереди восстановления;
- формирование Excel-отчетов;
- отправка отчетов, логов, backup-файлов и документов по импорту в Telegram;
- автообновление приложения через GitHub.

## Основной Бизнес-Процесс

Подробная пользовательская инструкция по ролям, шагам, ошибкам и диаграммам вынесена в [user-business-process-guide.md](/Users/anton/Documents/work/TakSklad/docs/user-business-process-guide.md).

Основной складской процесс:

1. Импорт Excel-документа через приложение или Telegram-бот.
2. Проверка и нормализация строк.
3. Сохранение метаданных импорта.
4. Группировка строк по клиенту, типу оплаты и адресу.
5. Сканирование КИЗов.
6. Контроль дублей и недосканированных позиций.
7. Сохранение результатов сканирования.
8. Печать сводных листов.
9. Отправка отчетов и уведомлений.
10. Хранение истории для разбора ошибок и аналитики.

Группировка обязательно должна учитывать клиента, тип оплаты и адрес. Это не дает смешивать в одну группу заказы одного клиента на один адрес, если у них разные типы оплаты.

## Принципы Надежности

TakSklad должен защищать складскую работу от типовых сбоев:

- временные ошибки Google Sheets;
- пропадание интернета;
- сбои принтера;
- аварийное закрытие приложения;
- дубли КИЗов;
- неполные импорты;
- отсутствующие или неправильные локальные настройки.

Для этого приложение должно использовать локальные очереди и backup-файлы, объяснять ошибки понятным языком и оставлять возможность восстановить работу без потери данных.

## Роль Telegram

Telegram - это рабочий канал управления, пока локальное приложение TakSklad запущено.

Текущие и будущие возможности Telegram:

- отчеты по отсканированным КИЗам;
- отчеты по импортированным документам;
- импорт Excel-документов, отправленных пользователями из разрешенных `chat_ids`;
- сегодняшний лог;
- уведомления о критических ошибках;
- уведомления после импорта;
- доставка backup-файлов и файлов восстановления;
- статус смены и прогресс сканирования;
- очереди ошибок и списки недосканированных позиций.

Если Telegram-бот должен работать 24/7 без открытого desktop-приложения, эту ответственность нужно переносить в отдельный backend-сервис.

## Данные И История

Экосистема должна постепенно централизовать:

- импортированные документы;
- строки импортов;
- отсканированные КИЗы;
- pending-очереди;
- отчеты;
- логи;
- backup-файлы;
- ошибки;
- действия пользователей;
- историю документов и смен.

Централизованная история позволит строить аналитику, аудит, веб-панели и автоматические уведомления.

## Будущая Архитектура

Проект может постепенно разделиться на несколько частей:

- локальное desktop-приложение для сканирования и печати;
- backend API для централизованных данных и 24/7-процессов;
- Telegram bot service для уведомлений и удаленных действий;
- веб-панель для менеджеров и администраторов;
- фоновые задачи для отчетов, очистки, backup и мониторинга.

Развитие должно быть постепенным. Локальное приложение должно оставаться стабильным и полезным, пока вокруг него появляются новые сервисы.
