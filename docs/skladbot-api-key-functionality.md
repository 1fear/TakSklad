# SkladBot WMS API Key Functionality Audit

Дата проверки: 23.05.2026  
Base URL: `https://api.skladbot.ru/v1`  
Формат авторизации: `Authorization: Bearer <API_KEY>`  

## Краткий вывод

API-ключи успешно авторизуются и дают широкий доступ на чтение к данным WMS SkladBot:

- клиенты;
- компании;
- склады;
- короба;
- продукты и остатки;
- складские движения;
- номинальные движения;
- заявки;
- календарь;
- финансовые документы;
- счета;
- отчёты по движениям, остаткам и счетам;
- справочники типов заявок, этапов, исполнителей, номенклатуры товаров и услуг.

По результатам тестов ключ выглядит как ключ чтения/отчётности. Документированные публичные endpoint'ы создания или редактирования заявок в API отсутствуют. Дополнительно проверены возможные create/update маршруты на тестовом клиенте `ИП Даврон`; они вернули `404` или `405`, то есть через публичный API создать или изменить заявку стандартными маршрутами не получилось.

Ключ также может обращаться к endpoint'ам генерации файлов, например PDF-этикеток или PDF-штрихкодов коробов, но это не изменение складских данных. Это скорее read/export-функциональность.

Важно: токены были переданы в чат, поэтому их лучше перевыпустить после завершения интеграции.

## Проверенные ключи

Были проверены два API-ключа. Оба ключа авторизуются и возвращают одинаковый уровень доступа на чтение.

JWT claims указывают:

- `sub`: `7877`;
- `scopes`: пустой список;
- фактические права определяются сервером SkladBot, а не видимыми OAuth scopes.

Сами токены в документ не записаны.

## Ограничения API

### Rate limit

При большом количестве быстрых запросов API возвращает:

```text
429 Too Many Attempts.
```

Для нормальной интеграции нужно:

- делать паузы между запросами;
- использовать пагинацию аккуратно;
- кешировать справочники;
- повторять запрос после паузы при `429`;
- не запускать много параллельных выгрузок.

### Форматы дат

Разные endpoint'ы используют разные форматы:

- `/warehouse/transactions`: `YYYY-MM-DD`;
- `/finance/documents`: `YYYY-MM-DD`;
- `/finance/invoices`: `YYYY-MM-DD`;
- `/calendar`: `YYYY-MM-DD`;
- `/report/transactions`: принимает `YYYY-MM-DD HH:MM:SS`;
- `/report/invoices`: по документации похож на DateTime, но фактически запрос с `YYYY-MM-DD HH:MM:SS` вернул ошибку формата; нужно использовать `YYYY-MM-DD`.

## Авторизация

Все рабочие endpoint'ы требуют заголовок:

```http
Authorization: Bearer <API_KEY>
Accept: application/json
Content-Type: application/json
```

Если токен неверный или истёк, сервер возвращает:

```json
{
  "message": "Unauthenticated."
}
```

## Что можно встроить в TakSklad

## 1. Выгрузка остатков клиента

Можно добавить модуль SkladBot, который:

- выбирает клиента из SkladBot;
- получает текущие товары через `/products`;
- фильтрует по SKU/артикулу/названию;
- выгружает текущий остаток;
- отправляет Excel в Telegram.

## 2. Остатки по дням

Можно делать расчёт:

1. получить текущий остаток по SKU через `/products`;
2. получить движения за период через `/warehouse/transactions`;
3. агрегировать движения по дате и SKU;
4. восстановить остаток на конец каждого дня;
5. сформировать Excel.

Именно так была сделана выгрузка:

```text
ООО Bastion Import Chapman MCHJ
Brown / Red / Gold
01.05.2026 - 23.05.2026
```

## 3. Сверка TakSklad с WMS

Можно сравнивать:

- что отсканировано в TakSklad;
- что списано/отгружено в SkladBot;
- расхождения по SKU;
- расхождения по количеству;
- заявки, где TakSklad закрыл документ, но WMS не видит движение;
- движения WMS, которых нет в TakSklad.

## 4. Telegram-бот для WMS

Можно добавить кнопки:

- остатки клиента сейчас;
- остатки клиента по дням;
- движения клиента за период;
- список заявок клиента;
- карточка заявки;
- финансы клиента;
- счета клиента;
- скачать Excel.

## 5. Автоотчёты

Можно настроить:

- ежедневный отчёт по остаткам;
- ежедневный отчёт по движениям;
- отчёт по клиенту;
- отчёт по SKU;
- отчёт по расхождениям между Google Sheets/TakSklad/SkladBot.

## 6. Ограничения для write-интеграции

Создание и редактирование заявок через текущий публичный SkladBot API не подтверждено.

Чтобы встроить создание заявок из TakSklad, нужно одно из двух:

1. получить у SkladBot отдельную документацию или API endpoint для создания заявок;
2. использовать не публичный API, а внутренние web-запросы личного кабинета, если это допустимо по правилам SkladBot.

Второй вариант технически возможен только после отдельного анализа web-интерфейса, но он менее надёжный: такие маршруты могут измениться без предупреждения.

## Практический вывод для проекта

Текущий API-ключ уже полезен для полноценной аналитики:

- остатки;
- движения;
- отчёты;
- финансы;
- заявки;
- сверки;
- Excel-выгрузки;
- Telegram-отчёты;
- расчёт исторических остатков по дням.

Но для управления WMS из TakSklad, например создания или редактирования заявок, текущего публичного API пока недостаточно. Нужно запросить у SkladBot write API для заявок или подтвердить скрытые endpoint'ы личного кабинета.
