## SkladBot token pool failover

- Причина: SkladBot номера снова подтягивались медленно из-за `429` от API.
- Дополнительно: Google-export мог зависнуть в `busy`, потому что session-level PostgreSQL advisory lock оставался на idle pooled connection после `commit`.
- Дополнительно по Google `429`: очередь теперь останавливается на первом rate limit и не переводит пачку событий в `failed`.
- Сделано:
  - добавлена поддержка `SKLADBOT_API_TOKENS` как пула токенов;
  - при `429` worker переключается на следующий токен, при `401/403` отключает плохой токен до перезапуска;
  - при временном `5xx` от SkladBot API worker делает короткую паузу перед повтором;
  - значения токенов не логируются и не записываются в ошибки;
  - на VDS установлен пул из `10` токенов;
  - `SKLADBOT_DETAIL_LIMIT=3` оставлен;
  - `SKLADBOT_REQUEST_DELAY_SECONDS` снижен с `20` до `2`;
  - PostgreSQL advisory lock для Google-export убран, pending-события защищаются row-level `FOR UPDATE SKIP LOCKED`;
  - Google pending export при `429/quota/rate limit` возвращает событие в `pending` и останавливает batch до следующего цикла;
  - пересобран `skladbot-worker`.
- Restore point:
  - `/opt/taksklad/restore_points/pre-skladbot-token-pool-20260603T141057Z`.
